# Module 3.2 - Data Security Risks and Mitigation

## 1. Business Impact of Data Security

* **Supports Compliance and Reputation:** Protecting data ensures compliance with regulatory requirements (e.g., GDPR, HIPAA). Data breaches erode customer trust and brand reputation.
* **Enables Business Continuity:** Securing critical data helps organizations recover quickly from disasters, ransomware, or insider threats. It significantly reduces downtime and financial loss.

---

## 2. Data Security Risks

| Threat | Description |
| :--- | :--- |
| **Hardware Failure** | Unexpected device failure can result in data loss or system downtime without proper backups. |
| **Malware** | Malicious software can encrypt, delete, or exfiltrate sensitive data, often spreading undetected. |
| **Data Corruption** | Software bugs, power loss, or faulty storage can alter data integrity, making it unusable or inaccurate. |
| **Insider Threats** | Employees or contractors may intentionally or accidentally misuse access to leak, alter, or destroy data. |
| **Theft (Unwanted Duplication / Publication)** | Stolen data may be secretly copied or publicly released, leading to IP loss, fraud, or reputational damage. |
| **Unwanted Access (Backdoors / Social Engineering)** | Attackers may exploit hidden system entry points or trick users into revealing credentials. |
| **Breaches (Identification / Disclosure)** | Data breaches expose personal or sensitive information, violating compliance laws and eroding trust. |

---

## 3. Strategies for Reducing Data Risk

| Control | Description |
| :--- | :--- |
| **Data Monitoring** | Continuously tracks data access and behavior to detect anomalies, misuse, or unauthorized activity in real time. |
| **Log Analysis (SIEM)** | Aggregates and analyzes system logs to identify patterns, detect threats, and support compliance through centralized visibility. |
| **Two-Person Integrity (TPI)** | Requires two individuals to authorize critical actions, reducing the risk of insider abuse or unilateral compromise. |
| **Split Encryption Keys / Tokens** | Distributes parts of cryptographic credentials across multiple individuals or systems to prevent single-point control. |
| **Separation of Roles** | Assigns distinct responsibilities (e.g., admin vs. auditor) to different users to enforce accountability and reduce the risk of privilege misuse. |

---

## 4. Regulatory Constraints on Data

| Category | Description |
| :--- | :--- |
| **Governmental Regulations** | Laws like GDPR, HIPAA, or FISMA mandate how organizations must protect, store, and process sensitive data, with penalties for non-compliance. |
| **Individually Privileged Information** | Includes legal, medical, or financial records protected by confidentiality laws (e.g., attorney-client privilege, doctor-patient confidentiality). |
| **Personally Identifiable Information (PII)** | Refers to data that can uniquely identify an individual (e.g., name + SSN); must be protected under laws like GDPR and CCPA. |
| **Payment Card Industry Data Security Standard (PCI DSS)** | An industry-mandated standard requiring businesses that handle credit cards to secure cardholder data through encryption, access control, and monitoring. |

---

## 5. Legal Considerations for Data

| Legal Consideration | Description |
| :--- | :--- |
| **Data Retention Requirements** | Organizations must retain certain types of data (e.g., financial, medical, communication logs) for legally defined periods to meet regulatory and audit obligations. |
| **Litigation Hold / Legal Hold** | When litigation is anticipated, businesses must suspend data deletion and preserve relevant records to avoid legal penalties. |
| **Subpoenas** | A legal order requiring an organization to produce specific data or records; failure to comply can result in fines or sanctions. |
| **Chain of Custody** | When data is involved in legal proceedings, it must be carefully tracked and documented to ensure it hasn't been altered or tampered with. |
| **Destruction Policies** | After legal and regulatory retention periods expire, data should be securely destroyed in accordance with written policies to reduce liability. |

---

## 6. Data Security Best Practices

* **Prioritize Data by Value:** Not all data is equal—focus protection on high-value assets like customer records, financials, and IP. Classifying data helps allocate security resources effectively.
* **Manage the Data Lifecycle:** Secure data from creation to deletion (e.g., storage, access, transfer, archival, disposal). Enforce retention policies and secure disposal to reduce liability and storage costs.
* **Balance Risk vs. Cost:** Stronger security typically increases cost, but ignoring risk can lead to breaches, fines, and downtime. Use risk assessments to determine where investment is most critical.
