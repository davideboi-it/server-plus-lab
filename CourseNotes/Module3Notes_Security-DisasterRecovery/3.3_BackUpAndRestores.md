# Module 3.3 - Backups and Restores

## 1. Backup Types Overview
Backups protect against hardware failures, data loss, corruption, and disasters (manmade or natural). Destinations can include local disks, removable media, network shares, and the cloud.

### Full Backup
* **Description:** The most basic and complete type of backup. Backs up all selected data to another set of media.
* **Archive Bit:** Changes (clears) the file's archive bit.
* **Performance:** Longest backup time, but the shortest restore time.
* **Requirement:** Provides a foundation for all other backup types.

### Differential Backup
* **Description:** Copies all data changed *since the last full backup*. Can be thought of as a "running backup."
* **Archive Bit:** Does NOT change the archive bit.
* **Performance:** Takes longer each day as the week goes by (files get larger until the next full backup). Restore takes less time than incremental (requires the Full backup + the latest Differential).

### Incremental Backup
* **Description:** Copies only the data that has changed *since the last full or incremental backup*.
* **Archive Bit:** Changes (clears) the archive bit.
* **Performance:** Fastest type of backup (takes up less storage space), but the longest restore time. 
* **Restore Process:** Requires the Full backup + *every* subsequent Incremental backup in chronological order.

### Snapshot
* **Description:** An image of the state of a Virtual Machine (VM) at a specific point in time. Used to restore a VM to a previous state.
* **Usage:** Typically requires the base image plus the snapshot(s). It should *not* be your only backup solution.

### Synthetic Full Backup
* **Description:** Combines an existing full backup and subsequent incremental backups to create a new, updated full backup file.
* **Benefits:** Reduces load on production systems (no need to re-read or copy unchanged files from the source). Faster backup creation and improves restore speed (no need to reassemble multiple incrementals during recovery).

### Archive
* **Description:** A copy of data stored for long-term retention (compliance, legal hold/e-discovery, historical purposes).
* **Usage:** Not intended for immediate operational recovery. Rarely accessed unless required for audits or litigation. Often stored on low-cost, high-capacity media (Tape, Optical discs, Cloud cold storage like AWS Glacier).

## 2. Open File Backups
Allows the backup of files that are actively being used or modified (e.g., live databases, Exchange email systems, VMs, continuously updated logs).
* **Benefit:** Prevents "file in use" errors and avoids data corruption, ensuring complete backups for 24/7 systems.
* **How it works:** Creates a point-in-time snapshot of the volume. The backup is performed on the snapshot, not the live file, allowing the application to keep running.
    * **Windows:** Uses Volume Shadow Copy Service (VSS).
    * **Linux/macOS:** Uses snapshot-based backups on the filesystem or hypervisor.

## 3. OS-Specific Backup Tools

### Windows Backups
* **Tools:** Windows Server Backup, `wbadmin` (CLI), Task Scheduler, PowerShell, or 3rd-party tools (Veeam, Acronis).
* **Example:** `wbadmin start backup -backupTarget: D: -include: C: -allCritical -quiet`

### Linux Backups
* **Tools:** Built-in tools like `tar`, `rsync`, `dd`, or `cpio` automated with `cron` jobs. Advanced tools include Bacula, Amanda, Timeshift, or `rsnapshot`.
* **Important:** Ensure permissions and SELinux contexts are preserved during backup.
* **Example:** `rsync -av /home/winnie/Downloads/ /tmp/backup`

### macOS Backups
* **Tools:** Time Machine (automated hourly/daily/weekly backups), `tmutil` (CLI), Carbon Copy Cloner, iCloud (for user files, not full system).
* **Example:** `tmutil startbackup --destination /Volumes/BackupDrive --block`

## 4. Backup Frequency Strategies

| Strategy | Description | Pros | Cons | Common Use Cases |
| :--- | :--- | :--- | :--- | :--- |
| **Daily Full** | A complete backup of all data is performed every day. | Simple recovery; no need to combine multiple backup sets. | High storage use and backup time. | Small environments with manageable data volumes. |
| **Weekly Full + Daily Incremental** | Full backup weekly; only changes since last backup are saved daily. | Efficient storage and faster daily backups. | Slower recovery (must restore full + all increments). | Most common enterprise strategy; balances time and storage. |
| **Weekly Full + Daily Differential** | Full backup weekly; daily backup includes all changes since last full. | Faster recovery than Incremental. | Daily backups grow larger over time. | Environments where restore speed is prioritized. |
| **Continuous Data Protection (CDP)** | Every change is backed up in near real-time. | Minimal data loss; very fast point-in-time recovery. | Requires specialized software; high resource use. | Critical systems like databases, financial apps. |
| **GFS (Grandfather-Father-Son)** | Rotational: Daily (Son), Weekly (Father), Monthly (Grandfather) full backups. | Long-term retention with structured recovery points. | Complex rotation and storage planning. | Environments with regulatory retention needs. |
| **Workload-Based Frequency** | Backup frequency is based on system role and data volatility. | Optimized for each server or app; avoids over/under-backing up. | Requires analysis and tuning; may lack consistency. | Hybrid infrastructures; mixed workloads. |
| **Frequent Local + Daily/Weekly Cloud Sync** | Frequent local snapshots with periodic cloud backup for redundancy. | Fast local recovery + offsite resilience. | Requires bandwidth planning and sync management. | Small businesses; cost-effective DR strategy. |

**Best Practices:** Define RPO (Recovery Point Objective) and RTO (Recovery Time Objective), test restore procedures regularly, monitor logs for failures, and automate schedules.

## 5. File-Level vs. System-State Backup

| Aspect | File-Level Backup | System State Backup |
| :--- | :--- | :--- |
| **Definition** | Backs up individual files and folders. | Backs up critical system files and configuration data. |
| **Purpose** | Restore user data or specific files. | Restore OS, roles, and system settings. |
| **Includes** | Documents, media, spreadsheets, app files. | Registry, boot files, AD, COM+, etc. |
| **Backup Size** | Smaller or variable. | Typically compact but essential. |
| **Restore Scope** | File-by-file. | Full system recovery or role-based restore. |
| **Common Use Cases**| Accidental file deletion, ransomware recovery. | Domain controller restore, OS recovery, disaster recovery. |

## 6. Backup Media

### Media Type Comparison
| Feature | Disk | Tape | Cloud | Print |
| :--- | :--- | :--- | :--- | :--- |
| **Speed** | Fast read/write | Slower access (especially restores) | Depends on bandwidth | Very slow (manual) |
| **Cost per GB** | Moderate | Low (great for large archives) | Subscription-based, scalable | Low storage cost, high labor |
| **Storage Capacity** | High (terabytes) | Very high (up to 18TB per LTO-9) | Virtually unlimited | Very low |
| **Longevity** | 3-10 years | 10-30 years (well-maintained) | Vendor-managed | Decades if stored well |
| **Security** | Encryptable, local | Encryptable, offsite | Encrypted at rest/transit | Physical protection only |
| **Disaster Resilience**| Moderate (onsite) | High (offsite) | Very high (geo-distributed) | Poor |
| **Best Use Case** | Fast local restores | Long-term archival or compliance | Disaster recovery, remote backup | Legal docs, offline TPM keys |

### Backup Media Inventory & Rotation
* **Inventory:** A comprehensive record of all media (tapes, disks, cloud) to support fast recovery, track movements, and flag missing or aging media.
* **Rotation:** Regularly switching storage media to prevent wear and support retention policies.
    * **FIFO (First In, First Out):** Oldest media reused first.
    * **GFS (Grandfather-Father-Son):** Balances short and long-term retention.
    * **Tower of Hanoi:** Complex algorithm spreading writes across fewer tapes to minimize wear.
    * **6-Tape or 10-Tape:** Fixed set of labeled tapes reused in a known cycle.

## 7. Backup Validation & Restores

### Validating Backups
Ensures backups are complete, functional, and restorable. A "successful backup" log does not guarantee usable data.
* **Goals:** Verify restore integrity (no corruption), meet compliance standards, ensure DR readiness, and protect against silent failures (bit rot, bad sectors).
* **Practice:** Test backups on alternate hardware to ensure recovery if original hardware fails.

### Restore Methods
| Method | Description | Use Case | Benefits | Drawbacks |
| :--- | :--- | :--- | :--- | :--- |
| **Overwrite** | Replaces existing files/system state with backup. | Full system recovery or data rollback. | Quick restore, original path preserved. | Risk of overwriting newer data. |
| **Side-by-Side** | Restores data to a separate directory on the same system. | Comparing versions, manual recovery. | No data is overwritten. | Requires extra space and manual intervention. |
| **Alternate Path** | Restores data to a different system or location. | Disaster recovery, testing, sandbox restore. | Preserves original system; great for testing. | May require reconfiguration of paths/permissions. |

**Restore Best Practices:**
* Perform periodic full restores in a test environment.
* Automate checksum/hash comparisons to detect corruption.
* Maintain spare equipment or cloud targets for rapid recovery.
