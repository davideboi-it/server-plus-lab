# Module 3.1 - Data Security Mechanisms

## 1. Data States
You can encrypt data in any of these states to increase confidentiality and trust:
* **Data at Rest:** Stored on a hard drive, USB stick, CD/DVD, or any other type of electronic storage medium.
* **Data in Transit:** Data is actively being transmitted on a network.
* **Data in Use:** Data is loaded into memory and is, or will shortly be, processed by the CPU.

---

## 2. Cryptography
Cryptography involves transforming Plain Text (unencrypted data) using a Key and a Cipher (encryption algorithm) to produce Ciphertext (encrypted data).

### Symmetric Encryption
* The same key is used to encrypt and decrypt the data.
* Used extensively to protect data at rest and provides confidentiality.
* Excellent for bulk data encryption because it is fast and has good performance.
* **Risks:** You must share the key in advance; if the single key is compromised, all files are at risk of losing confidentiality.

### Asymmetric Encryption
* Uses two mathematically related keys: a Public Key and a Private Key.
* You encrypt with the Public Key and decrypt with the Private Key.
* Provides both confidentiality and integrity.
* **Best Practices:** You can freely distribute your Public Key, but you must carefully guard the Private Key, never letting anyone else access it, and keep it stored in an encrypted format.

---

## 3. Hashing
* A hashing function takes data of any size or type and creates a fixed-length string of characters (called hash values, hash codes, digests, or hashes).
* It creates a one-way encryption that does not require a key and does not modify the original file/data.
* The slightest change to the input dramatically changes the output.
* **Common Uses:** Securely storing passwords and verifying the integrity of files or packets.

---

## 4. Data Retention Policies
A data retention policy is an organization's formal guideline for what data is stored, how long it is stored, and when/how it is deleted, archived, or destroyed.

* **Key Goals:** Ensure compliance with legal, regulatory, and contractual requirements, support business continuity and disaster recovery, minimize storage costs, and protect sensitive data.
* **Best Practices:** Align with industry regulations (e.g., HIPAA, GDPR, SOX), consult the legal department, automate retention and disposal, regularly review the policy, and enforce auditable processes.

### Core Components
| Component | Description |
| :--- | :--- |
| **Data Categories** | Defines types of data (e.g., HR records, financials, emails, logs). |
| **Retention Periods** | Specifies how long each data type is kept (e.g., 7 years for tax records). |
| **Storage Locations** | On-site servers, cloud storage, tape archives, etc. |
| **Archiving Rules** | When to move data from primary to long-term storage. |
| **Disposal Procedures** | Secure deletion or destruction methods after the retention period expires. |
| **Access Control** | Who can access retained data and under what conditions. |
| **Legal Hold Exceptions** | Overrides for data under litigation or investigation. |

---

## 5. Data Storage: On-Site vs. Off-Site
The physical location of stored data impacts security, disaster recovery, accessibility, compliance, and cost.

| Feature | On-Site Storage | Off-Site Storage |
| :--- | :--- | :--- |
| **Location** | On company premises (server room, data center). | Remote facility, cloud provider, or secondary site. |
| **Access Speed** | Fast local access. | Slower (depends on internet or WAN connection). |
| **Disaster Resilience** | Vulnerable to local disasters. | Resilient to local outages or physical damage. |
| **Control** | Full control over hardware and policies. | Shared responsibility with third-party vendors. |
| **Security** | Requires strong physical and network security. | Requires strong encryption and secure transport. |
| **Scalability** | Limited by local hardware and space. | Highly scalable (especially cloud-based). |
| **Cost** | High upfront (CAPEX), lower ongoing (OPEX). | Pay-as-you-go (OPEX), potentially lower upfront. |
| **Compliance** | Easier to meet internal policy requirements. | Must ensure provider meets regulatory standards. |
| **Backup/Restore Speed** | Fast for large restores. | Slower, especially for full system recovery. |
| **Maintenance** | Handled in-house. | Managed by vendor or remote IT staff. |
