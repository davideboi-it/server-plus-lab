# Module 3.5 - IAM for Server Administration

## 1. Identity and Access Management (IAM) Overview
IAM controls who can access servers, what they can do, and how their activity is monitored.
* **Key Benefits:**
    * Enforces **least privilege** (users get only the access they need).
    * Prevents unauthorized access to sensitive server functions.
    * Audits and tracks administrative actions for accountability.
    * Reduces insider threat risk and prevents lateral movement in attacks.
    * Supports compliance with frameworks like HIPAA, PCI-DSS, and NIST.

### Core IAM Components
| Component | Function |
| :--- | :--- |
| **Authentication** | Verifies user identity (e.g., password, smart card, MFA). |
| **Authorization** | Grants or denies actions based on roles (e.g., admin vs. read-only). |
| **Account Management** | Controls account creation, deactivation, expiration, and role assignment. |
| **Privileged Access** | Restricts powerful admin actions to authorized personnel only. |
| **Audit Logging** | Tracks login attempts, changes, and administrative actions. |

### Common IAM Tools
* **RBAC / ABAC:** Role-Based or Attribute-Based Access Control.
* **MFA:** Multi-Factor Authentication for all admin accounts.
* **JIT (Just-In-Time) Access:** Temporary elevated access.
* **Directory Services:** Active Directory, LDAP.
* **Centralized IAM Platforms:** Microsoft Entra ID, FreeIPA, Okta.

## 2. User Accounts & Policies

### Users and Groups
* **Users:** Individual accounts assigned to people or services for system access.
* **Groups:** Simplify permissions by assigning access rights to many users at once.
* **Best Practice:** Admins should avoid assigning permissions directly to individual users. Always use group-based access control to enforce consistent privileges and support least privilege.

### Password Policies
* Enforce minimum length, complexity, and expiration to strengthen security.
* Use **account lockout policies** to prevent brute-force attacks (e.g., lock account for 30 minutes after 5 failed logon attempts).
* Encourage or require MFA for critical accounts.
* Do not reuse passwords across systems or share credentials between users.

## 3. Permissions and Access Control Models

* **Role-Based Access Control (RBAC):** Access is granted based on a user's role (e.g., Admin, HR, Developer) implemented as user groups. Widely used for file, print, and database permissions.
* **Rule-Based Access Control:** Access determined by system-enforced rules (time of day, location, device type), not user identity. Common in firewalls and contextual access.
* **Scope-Based Access Control:** Access is restricted based on defined boundaries (departments, projects, tenants). Often used in cloud environments and APIs with authorization tokens (OAuth).
* **Segregation of Duties (Separation of Duties):** Separates critical tasks across multiple people to reduce risk of fraud or abuse. No single person controls an entire sensitive process (e.g., one creates accounts, another approves them).
* **Delegation:** Allows trusted users to perform specific admin tasks (like helpdesk resetting passwords) without giving them full global privileges.

## 4. Auditing

Auditing involves recording and reviewing system events to ensure accountability, compliance, and security.

### Key Audit Targets
| Audit Target | Purpose |
| :--- | :--- |
| **User Activity** | Tracks specific config or service changes. Supports forensics. |
| **Logins & Authentication** | Records successes/failures. Detects brute-force and unauthorized access. |
| **Group Membership Changes** | Logs who was added/removed to control privileged access. |
| **Object Access & Deletion** | Audits files/resources. Identifies leaks, tampering, or loss. |
| **Privilege Use** | Tracks use of elevated tools to detect privilege escalation. |

### Typical Server Auditing Tools
* **Event Viewer / Security Logs:** Core Windows tool (e.g., Event 4624 for logon, 4720 for user creation).
* **Audit Policy & GPMC:** Configure what events are logged across domains.
* **PowerShell:** Scripted log access (`Get-WinEvent`).
* **Sysmon (Sysinternals):** Advanced logging of processes, network, and files.
* **SIEM Tools:** Splunk, Graylog, ELK, MS Sentinel for aggregating and analyzing logs.

## 5. Multi-Factor Authentication (MFA)

MFA requires two or more *different* factor types to grant access.

### Authentication Factors
* **Something You Know (Knowledge):** Password, PIN, security questions, pattern lock.
* **Something You Have (Possession):** Smart card, hardware token (RSA SecurID), authenticator app (Google/Microsoft), SMS/Email OTP, USB key (YubiKey).
* **Something You Are (Inherence/Biometric):** Fingerprint, facial/retina scan, voice recognition.
* **Somewhere You Are (Location):** Geolocation (IP/GPS), time zone restrictions, approved networks.
* **Something You Do (Behavioral):** Typing rhythm, mouse movement patterns.

---

## 6. Single Sign-On (SSO)

An authentication process allowing a user to log in once to a central Identity Provider (IdP) and gain access to multiple systems without re-entering credentials. (e.g., SAML, OAuth, OpenID Connect).

* **Benefits:** Improves user experience, reduces password fatigue and support tickets, centralizes management and auditing.
* **Considerations:** If the SSO provider is compromised, all linked systems are at risk. Strong MFA is mandatory.
