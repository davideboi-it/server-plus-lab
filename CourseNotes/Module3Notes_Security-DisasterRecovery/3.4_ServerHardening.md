# Module 3.4 - Server Hardening Methods

## 1. Firmware and Boot-level Security Controls

### UEFI/BIOS Passwords
Passwords set within the firmware interface (UEFI or legacy BIOS) to control access to system-level settings.
* **Types:**
    * **Supervisor/Admin Password:** Protects access to BIOS/UEFI settings.
    * **User Password:** Prevents system from booting without authentication.
    * **HDD Password (optional):** Locks access to a specific drive at the firmware level.
* **Purpose:**
    * Prevents unauthorized users from changing boot order, disabling security features, or tampering with hardware-level settings.
    * Adds a layer of protection against physical tampering.
* **Considerations:**
    * If forgotten, may require motherboard jumper reset or contacting manufacturer.
    * Not encrypted—can be bypassed by skilled attackers with physical access.

### Bootloader Passwords
Passwords added to the bootloader configuration (e.g., GRUB on Linux) to restrict access to boot options or kernel parameters.
* **Common Use:**
    * **GRUB Passwords:** Require authentication to enter single-user mode or edit boot entries. Protects against attacks like kernel parameter tampering or bypassing login authentication.
* **Purpose:**
    * Prevents unauthorized users from booting into recovery or debug modes.
    * Helps enforce system integrity before the OS even loads.
* **Considerations:**
    * Should be combined with full-disk encryption for comprehensive protection.
    * Bootloader configs must be secured with correct file permissions.

### Firmware and Bootloader Password Comparison
| Feature | UEFI/BIOS Password | Bootloader Password |
| :--- | :--- | :--- |
| **Where Set** | Firmware interface | Bootloader config file (e.g., `/boot/grub/grub.cfg`) |
| **Protects** | BIOS/UEFI settings, boot sequence | Kernel parameters, recovery access |
| **Used On** | All OS types | Primarily Linux and Unix-based systems |
| **Security Role** | Prevents tampering before OS loads | Prevents bypassing OS security controls |

## 2. Operating System (OS) Hardening

### Strategies for OS Hardening
* Change defaults.
* Disable unnecessary services, protocols, and ports.
* Implement OS security policies.
* Implement strict access control based on "least privilege".
* Implement secure authentication, including MFA for admins.
* Deploy in a secure physical location.
* Configure the host firewall.
* Place in a secure VLAN/network segment.
* Monitor using network- and host-based IDS.
* Control traffic using a network firewall and/or IPS appliance.
* Implement change management.
* Patch regularly.
* Test patches and be able to roll back changes if necessary.
* Maintain backups.
* Where appropriate, implement high availability and/or load balancing key service(s).
* Forward logs to a syslog server.
* Preferably, continuously monitor with a SIEM.
* Have a disaster recovery plan with RTO and RPO for key services.

## 3. Application Hardening

Hardening reduces the attack surface and strengthens an application's resistance to exploitation.

### Application Hardening Techniques
* **Authentication & Access Control:**
    * Enforce strong authentication (e.g., MFA, complex passwords).
    * Role-based access control (RBAC) to limit privileges.
    * Account lockout policies to prevent brute-force attacks.
* **Code & Configuration Security:**
    * Input validation to prevent injection attacks (e.g., SQLi, XSS).
    * Output encoding/sanitization to block reflected code execution.
    * Disable or remove unused features, plugins, or services.
    * Minimize debug and verbose error messages in production.
* **Update & Patch Management:**
    * Regularly update libraries, frameworks, and dependencies.
    * Apply vendor patches as soon as they're available.
    * Monitor for end-of-life software and replace it.
* **Runtime & Execution Controls:**
    * Enable application whitelisting.
    * Use Address Space Layout Randomization (ASLR) and Data Execution Prevention (DEP).
    * Implement code signing for executables.
* **Logging & Monitoring:**
    * Enable application logging for authentication, errors, and unusual behavior.
    * Use SIEM tools to monitor logs for security events.
    * Alert on anomalies (e.g., access from unknown IPs).
* **Data Protection:**
    * Use encryption for sensitive data in transit and at rest.
    * Avoid storing unnecessary PII or secrets in code/configs.
    * Implement secure cookie flags (e.g., HttpOnly, Secure, SameSite).
* **Secure Development Practices:**
    * Follow secure coding standards (e.g., OWASP Top 10).
    * Perform code reviews and static/dynamic analysis.
    * Conduct regular penetration testing or vulnerability scanning.

## 4. Host Security

### Strategies for Hardening Workstations
* Change defaults.
* Disable unnecessary services, features, and protocols.
* Require strong authentication.
* Enable host firewall.
* Enable built-in OS and OEM security features.
* Install latest patches and security updates.
* Install/enable antivirus/anti-malware software.
* Regularly back up data.
* Prefer to deploy from a golden image.

### Hardening Hardware
* Physically secure hardware.
* Disable unneeded hardware (Physical ports, devices, or functions).
* Set BIOS password.
* Set boot order.

## 5. Patch Management
Patch management ensures systems stay secure, stable, and up to date by applying vendor-released fixes for vulnerabilities, bugs, and performance issues.

**Patch Management Process:**
1. Identify needed patches.
2. Test in a non-production environment.
3. Approve via change management.
4. Deploy to production systems.
5. Verify patch success and system stability.
6. Document the process and results.
