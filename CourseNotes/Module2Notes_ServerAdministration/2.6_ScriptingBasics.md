# 2.6 Scripting Basics for Server Administration

## 1. Why Use Scripting for Server Management?
Scripting shifts administration from manual, repetitive clicking to automated, predictable code execution.
* **Automation & Time Saving:** Automates routine tasks such as user creation, service monitoring, updates, and backups.
* **Consistency:** Standardizes how tasks are performed across multiple servers, drastically reducing human error.
* **Scheduling:** Enables unattended maintenance via tools like Windows Task Scheduler or Linux `cron`.
* **Configuration Management:** Makes it easy to rapidly apply, reapply, or deploy settings systematically.
* **Troubleshooting & Auditing:** Allows administrators to log outputs and track historical actions for easier debugging.

---

## 2. Common Server-Side Scripting Languages
Different environments require different automation tools.
* **PowerShell (`.ps1`):** The standard for Windows automation (AD, Hyper-V) but now cross-platform (PowerShell Core).
* **Bash (`.sh`):** Shell scripting used for startup scripts, file operations, and cron jobs in Linux/macOS environments.
* **Python (`.py`):** Cross-platform language heavily used for API integration, data parsing, and modern DevOps tools.
* **Batch (`.bat` / `.cmd`):** Legacy Windows command-line automation.
* **Ansible (YAML):** Agentless infrastructure automation and configuration management (treats infrastructure as code).
* **JavaScript (Node.js) & Groovy:** Used for API calls and CI/CD pipelines (like Jenkins).

---

## 3. PowerShell Core Concepts
PowerShell is drastically different from traditional text-based shells (like Bash or Command Prompt).
* **Object-Based Pipeline:** Instead of passing raw text strings between commands, PowerShell passes rich `.NET` objects. This means you can easily extract specific properties (like a service's RAM usage) without writing complex text-parsing scripts.
* **Cross-Platform:** Originally built for Windows, but can now be installed on Linux via package managers (`apt`, `yum`) and launched using the `pwsh` command. Scripts use the same `.ps1` extension.

---

## 4. Scripting Logic & Syntax (PowerShell Focus)
* **Variables:** Store data for later use. They always start with a `$` sign (e.g., `$serverName = "Server01"`).
* **Data Types:** Define the kind of data stored: `[int]` for numbers, `[string]` for text, `[array]` for lists.
* **Environment Variables:** System-level settings accessible by all processes, accessed using the `$env:` scope (e.g., `$env:USERNAME`, `$env:COMPUTERNAME`).
* **Comments:** Code ignored during execution, used for documentation. Starts with a `#`.
* **Comparison Operators:** Used to compare values: `-eq` (equal), `-ne` (not equal), `-gt` (greater than), `-lt` (less than).
* **Conditionals:** Control flow logic using `if`, `elseif`, and `else` blocks.
* **Loops:** Repeat actions systematically using `foreach`, `for`, or `while`.

---

## 5. Script Execution & Security
By default, Windows Server protects itself against malicious scripts by restricting execution.
* **Execution Policy:** A safety feature that determines which PowerShell scripts can run. 
    * To run locally written scripts, you typically must lower the restriction level by running: `Set-ExecutionPolicy RemoteSigned`.
    * *Exam Tip:* If a script throws a red error simply saying it cannot be loaded, the Execution Policy is usually set to `Restricted` (the default).

---

## 6. Remote Management (PowerShell Remoting)
PowerShell empowers remote administration over a network or cloud. This relies on **WinRM** (Windows Remote Management).
* **Prerequisites:** Computers should ideally be on the same domain or trusted network. Target server must have WinRM enabled.
* **Enable Remoting:** Run `Enable-PSRemoting -Force` on the target server to configure WinRM and firewall rules.
* **Workgroup Setup:** If not on a domain, the local managing machine must explicitly trust the remote server: `Set-Item WSMan:\localhost\Client\TrustedHosts -Value "RemoteServerNameOrIP"`.
* **Interactive Session:** Connect to a remote server's terminal exactly like SSH in Linux: 
    * `Enter-PSSession -ComputerName <IP> -Credential (Get-Credential)`.
    * To close the session, type `Exit-PSSession`.
